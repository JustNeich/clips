# Daily runbook

## 1. Один preflight

До создания роликов код проверяет одним проходом:

- защищённый credential доступен без сообщения владельцу;
- Clips profile ID и YouTube channel ID точно совпадают с `CHANNELS.md`;
- существующие workers здоровы, расписание читается, свободные slots известны;
- нет конфликтующего активного run;
- есть минимум шесть `ready`-источников на каждый канал;
- ledger доступен для resume и dedupe.

Если preflight не пройден, сначала запускается уже настроенный короткий repair/refill. Не делай deploy, migration, смену credential или channel binding внутри daily run.

## 2. Обработка

Цель — ровно три новых ролика на каждый канал, не три всего. Три channel workers работают параллельно; внутри одного ролика этапы последовательны. На один ролик допускается только один активный maker. Один общий judge может по очереди проверять однотипные результаты.

Модель используется только для source fit, caption/montage и смыслового QA. Счёт, schedule, состояние, retry, dedupe, upload reconciliation и polling ведёт обычный код. Используй уже работающий маршрут `gpt-5.6-luna`, если он настроен; не запускай подбор моделей во время выпуска.

Путь slot: reserve source → source maker → blind source judge → caption/montage maker → blind plan judge → render → объективный QA → blind final judge → schedule/upload → public verification.

- Обычная внешняя ошибка: до трёх коротких повторов.
- Если Visolix и worker-download не дают файл, runner берёт уже подготовленный MP4 или скачивает его локальным `yt-dlp`, загружает через существующий Clips source-upload и продолжает тот же QA. Это автоматический fallback, не новый source и не смена credentials.
- Semantic `REWORK`: максимум два, затем source `REPLACE`.
- Сломанный канал не останавливает два остальных.
- Во время ожидания slot и polling не вызывай модель.
- Во время daily run не меняй код, инфраструктуру или архитектуру.

Для первого production-подтверждения можно сначала провести по одному canary на канал. После их проверки выпускаются оставшиеся шесть.

## 3. Upload и публичность

При timeout или неизвестном upload сначала reconcile существующие upload/session данные и YouTube ID. Никогда не запускай слепой повтор, пока не доказано, что видео не создано.

Публикация считается подтверждённой только когда согласованы:

1. Clips publication state;
2. YouTube RSS нужного channel ID;
3. точная публичная `/shorts/<youtube_id>`-страница.

Только после этого запиши `public_verified`, сохрани YouTube ID и переведи source в `used`.

## 4. Завершение и продолжение

Не завершай при `8/9`. После `3/3/3` refiller восстанавливает минимум шесть `ready`-источников на канал. Сохрани фактические время, model calls, retries, replacements, ошибки, стоимость и следующий scheduled wake. Следующий wake должен начать обычный preflight без команды из чата.

Статус владельцу короткий: текущий этап, счёт по трём каналам, один реальный blocker, автоматический repair и следующий проверяемый результат. Не отправляй обычные промежуточные просьбы про token, слабый source, slots или retry.

## 5. Внешний blocker и безопасная остановка

При действительно внешнем blocker продолжай остальные каналы, создай один maintenance packet с точной причиной, evidence и минимальным ремонтом. Не ослабляй content/QA gates и не закрывай задачу до ремонта.

При `SIGTERM` перестань брать новые slots, сохрани этап каждого slot и доведи начатый upload до известного или reconciled состояния. Resume продолжает из ledger и не создаёт дубль.
