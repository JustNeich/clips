# Source Search gate: owner decision 2026-07-11

Статус: принято владельцем 2026-07-11 (~15:05 MSK). Управляемое изменение
критерия gate, оформленное явно.

## Проблема

После исправления границы роли (overlay v1–v3, см.
`source-search-role-boundary-*.overlay.json` и коммит 8ee599d) оба маршрута
gpt-5.6-luna стабильно проходят 28 из 30 кейсов. Два оставшихся кейса —
пограничные по концепту, и ответы модели на них нестабильны при
байт-в-байт одинаковых входах:

| Кейс | v3-run | v4-run | v5-run |
|---|---|---|---|
| source-search-09-DW0w8RMjY3Y (статичный стоп) | FOUND | NO_MATCH | FOUND |
| source-search-13-DWwSVVOjMqO (авария+интервью) | FOUND | FOUND | NO_MATCH |

(в v4 и v5 входы идентичны; расхождение — сэмплинг модели). Остальные 28
кейсов дали одинаковые ответы во всех прогонах.

С полом 30/30 прохождение gate становится лотереей (~25% на прогон,
60 вызовов за попытку) — критерий перестаёт быть воспроизводимым.

## Решение владельца

Пол точности source_search: **28/30** (реализован как 0.93 из-за округления
агрегатов до шести знаков; любой порог в (27/30, 28/30] отбирает одинаково).
Прочие требования без изменений: schema success = 1, p95 ≤ 300 с, полная
usage/cost-телеметрия, кандидаты — исключительно gpt-5.6-luna.

Обоснование:
1. Ошибки этой роли не имеют цены безопасности: лишний FOUND стоит один
   downstream Source Fit вызов (~0.5 кредита), лишний NO_MATCH — один
   пропущенный кандидат буфера (компенсируется refill'ом). Все опасные
   решения принимают downstream-этапы (Source Fit, Source Policy, Vision QA)
   со своими жёсткими gate.
2. Нестабильность двух пограничных кейсов доказана тремя прогонами; порог
   28/30 отделяет воспроизводимое ядро от сэмплингового шума, не ослабляя
   проверяемое качество.

## Следствие

`real-30-v5-search-boundary`: luna/low 28/30 и luna/medium 28/30 — оба
проходят; primary выбирается по цене, второй уровень — same-route fallback.
Пере-выпуск selection-evidence выполняется чистым replay из checkpoint без
новых вызовов моделей.
